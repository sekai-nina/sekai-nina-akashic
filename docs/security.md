# セキュリティ設計

Akashic のセキュリティは **3つの仕組み** で成り立っています。

## 1. 認証（あなたは誰？）

ユーザーがログインする仕組み。2つの方法があります。

### Discord ログイン

Discord アカウントでログインできます。Discord のアイコンや名前がそのまま使われます。

### メール + パスワード

従来のメール・パスワードでもログインできます。

### 二段階認証（MFA）

**全ユーザー必須** です。初回ログイン後、Google Authenticator 等のアプリで QR コードをスキャンして設定します。以降、ログインのたびに 6 桁のコードを入力する必要があります。

パスワードが漏れても、スマホがなければログインできません。

### 招待制

アカウントは **管理者が発行した招待リンク** からのみ作成できます。知らない人が勝手にアカウントを作ることはできません。

招待リンクには以下が設定されています：
- **ロール**（管理者 / メンバー / 閲覧者）— 何ができるか
- **クリアランスレベル**（一般 / 限定 / 極秘）— 何が見えるか
- **有効期限** — 期限切れになると使えない
- **1回限り** — 同じリンクで2人が登録することはできない

## 2. 機密レベル（何が見える？）

アセット（データ）には **機密レベル** が設定されています。ユーザーには **クリアランスレベル** が設定されています。

```
クリアランスが高い人は、低いレベルのデータも全て見えます。

  極秘    → 極秘 + 限定 + 一般 が見える
  限定    → 限定 + 一般 が見える
  一般    → 一般 だけ見える
```

### 具体例

| アセット | 機密レベル | 一般ユーザー | 限定ユーザー | 極秘ユーザー |
|---------|-----------|:---:|:---:|:---:|
| ブログ記事 | 一般 | 見える | 見える | 見える |
| 非公開写真 | 限定 | **見えない** | 見える | 見える |
| 個人情報系 | 極秘 | **見えない** | **見えない** | 見える |

「見えない」は検索にも引っかからず、存在すら分かりません。

## 3. 防御の仕組み（どう守っている？）

### 二重の壁

```
ユーザー → [アプリの壁] → [データベースの壁] → データ
```

#### 壁1: アプリの壁（withClearance）

アプリのコードが「このユーザーのクリアランスは○○」とデータベースに伝えてからデータを取得します。伝え忘れた場合、**データベースは何も返しません**（安全側に倒れる）。

#### 壁2: データベースの壁（PostgreSQL RLS）

データベース自体に「このクリアランス以下のデータしか返さない」というルールが組み込まれています。アプリにバグがあっても、データベースが最後の砦として守ります。

### フェイルセーフ（壊れたときにどうなるか）

| 何が起きた | 結果 |
|-----------|------|
| 開発者がフィルタを書き忘れた | データベースが何も返さない（安全） |
| アプリにバグがあった | データベースが何も返さない（安全） |
| クリアランスの設定を忘れた | データベースが何も返さない（安全） |
| 未知のクリアランスレベルが来た | データベースが何も返さない（安全） |

全て「壊れたら何も見せない」方向に倒れます。「壊れたら全部見える」には**なりません**。

### 保護対象テーブル

以下の 8 テーブルは機密レベルで保護されています：

| テーブル | 説明 | 保護の仕組み |
|---------|------|-------------|
| Asset | アセット本体 | 自身の機密レベルで判定 |
| AssetText | テキスト（ブログ本文等） | 親 Asset の機密レベルで判定 |
| AssetEntity | エンティティ紐付け | 親 Asset の機密レベルで判定 |
| AssetRelation | アセット間の関連 | 両端の Asset で判定 |
| SourceRecord | 出典情報 | 親 Asset の機密レベルで判定 |
| Annotation | 注釈 | 親 Asset の機密レベルで判定 |
| CollectionItem | コレクション内のアイテム | 親 Asset の機密レベルで判定 |
| Testimonial | 口コミ | 親 Asset の機密レベルで判定 |

その他のテーブル（User, Entity, Collection 等）は機密レベルによる保護対象外ですが、RLS によりログインしていない外部からのアクセスは全てブロックされています。
